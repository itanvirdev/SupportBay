# Purpose

Activity Logs provide a complete audit trail of important actions occurring within SupportBay.

Activities are automatically generated by the system whenever significant actions occur.

Activity logs are immutable records and should never be editable.

---

# Core Principle

Activities are not messages.

Messages represent communication.

Activities represent system actions.

---

## Example

Message:

Agent Reply

"Please clear your cache and try again."

---

Activity:

Ticket assigned to John Smith.

Priority changed from Normal to High.

Department changed from Pre-Sales to Technical Support.

---

# Visibility

## Customer

Can view customer-safe activities.

Examples:

- Ticket Created
- Ticket Closed
- Ticket Reopened

Cannot view:

- Internal assignments
- Internal notes
- Staff actions

---

## Agent

Can view activities for accessible tickets.

---

## Manager

Can view all activities.

---

## Administrator

Can view all activities.

Each activity contains:

- Activity Type
- Actor
- Related Entity
- Metadata
- Timestamp

# Database Table

Table:

wp_sbay_activities

---

## Columns

| Column        | Type            | Nullable | Description               |
| ------------- | --------------- | -------- | ------------------------- |
| id            | BIGINT UNSIGNED | No       | Activity ID               |
| activity_type | VARCHAR(100)    | No       | Event type                |
| actor_id      | BIGINT UNSIGNED | Yes      | User who triggered action |
| entity_type   | VARCHAR(50)     | No       | Related entity            |
| entity_id     | BIGINT UNSIGNED | No       | Related entity ID         |
| target_type   | VARCHAR(50)     | No       | Action: Assigned Ticket   |
| target_id     | BIGINT UNSIGNED | No       | Target: Agent #25         |
| ticket_id     | BIGINT UNSIGNED | Yes      | Related ticket            |
| metadata      | LONGTEXT        | Yes      | JSON metadata             |
| visibility    | VARCHAR(20)     | No       | Visibility level          |
| created_at    | DATETIME        | No       | Timestamp                 |

# Timeline Integration

Activities should appear in ticket timelines.

Example:

Customer created ticket

↓

Activity:
Ticket assigned to Agent

↓

Agent replied

↓

Activity:
Priority changed to High

↓

Customer replied

---

# Activity Retention

Activities should never be automatically deleted.

Reasons:

- Auditing
- Compliance
- Reporting
- Historical tracking

---

# Search & Filtering

Staff should be able to filter activities by:

- Activity Type
- User
- Ticket
- Date Range
- Department

# Notifications

Activities may trigger notifications.

Examples:

ticket_assigned
↓
Notify Assigned Agent

purchase_verified
↓
Notify Ticket Owner

# Security Rules

Activities are immutable.

No editing allowed.

No deletion allowed through UI.

Only administrators may purge activities manually.
