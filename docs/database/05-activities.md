# SupportBay – Database Specification

## Table: `wp_sbay_activities`

---

# Purpose

Stores an immutable audit trail of actions performed on tickets.

Activities are **system events**, not conversations.

They provide accountability, traceability, reporting, and historical context.

---

# Relationships

```text
Ticket
   │
   ▼
Activities
```

Each activity belongs to one ticket.

---

# Table Structure

| Column     | Type            | Null | Default           | Index   | Description                          |
| ---------- | --------------- | ---- | ----------------- | ------- | ------------------------------------ |
| id         | BIGINT UNSIGNED | No   | AUTO_INCREMENT    | PRIMARY | Activity ID                          |
| ticket_id  | BIGINT UNSIGNED | No   | -                 | INDEX   | Parent ticket                        |
| actor_id   | BIGINT UNSIGNED | Yes  | NULL              | INDEX   | WordPress user ID                    |
| actor_type | VARCHAR(20)     | No   | system            | INDEX   | customer, agent, manager, system, ai |
| event      | VARCHAR(50)     | No   | -                 | INDEX   | Event identifier                     |
| payload    | LONGTEXT        | Yes  | NULL              | -       | JSON event data                      |
| ip_address | VARCHAR(45)     | Yes  | NULL              | -       | IPv4 / IPv6 address (optional)       |
| user_agent | TEXT            | Yes  | NULL              | -       | Client user agent (optional)         |
| created_at | DATETIME        | No   | CURRENT_TIMESTAMP | INDEX   | Event timestamp                      |

# Payload

The payload stores structured JSON related to the event.

Examples:

### Ticket Assigned

```json
{
	"old_agent_id": null,
	"new_agent_id": 12
}
```

### Status Changed

```json
{
	"old_status": "open",
	"new_status": "resolved"
}
```

### Department Changed

```json
{
	"old_department_id": 2,
	"new_department_id": 5
}
```

---

# Business Rules

- Activities are immutable.
- Activities cannot be edited.
- Activities should not be manually deleted.
- Every activity belongs to one ticket.
- Activities are generated by services, not UI components.
- Activities are not customer conversations.

---

# Visibility

Customers:

- Cannot view activities.

Support Staff:

- Can view full activity timeline.

Administrators:

- Can access all activities.

---

# Timeline

Activities appear in chronological order.

Newest:

```text
ORDER BY created_at DESC
```

Oldest:

```text
ORDER BY created_at ASC
```

depending on UI context.

---

# Repository Responsibilities

Activity Repository handles:

- Logging events
- Retrieving timelines
- Filtering by event
- Filtering by actor
- Reporting support

Business logic belongs in the service layer.
